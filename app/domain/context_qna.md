# 컨텍스트: 질의회신(QNA) 기반 상황 매칭 체크리스트 작성

## 이 문서의 목적

K-IFRS 제1115호 관련 **질의회신(QNA) 사례**를 분석하여,  
꼬리질문 시스템에 사용할 **"상황 매칭형 체크리스트"**를 함께 만드는 것입니다.

**작업 결과물**: `app/domain/decision_trees.py` 파일의 `QNA_MATCH_TREES` dict 항목

---

## 배경: 본문 체크리스트와 무엇이 다른가

| 구분 | 본문/적용지침 체크리스트 | QNA 체크리스트 |
|------|------------------------|--------------|
| **목적** | 판단 요소를 하나씩 확인 | QNA 전제조건과 사용자 상황 일치 여부 확인 |
| **질문 성격** | "당신 거래에서 X는 누가 하나요?" | "당신 상황이 이 QNA와 같은 조건인가요?" |
| **출력** | 판단 요소 리스트 | 전제조건 체크리스트 |
| **근거** | 기준서 조항 (문단 번호) | 질의회신 ID + 전제 조건 |

---

## QNA 체크리스트의 역할

질의회신은 **특정 전제조건 하의 회계처리**를 다룹니다.  
사용자의 상황이 그 전제조건과 다르면, 같은 회계처리를 적용할 수 없습니다.

예시:
```
[QNA 원문]
질의: 반품 가능 기간이 6개월이고, 반품 시 현금 환불하는 소매업 매출의 수익 인식
회신: 변동대가로 처리하고, 기댓값으로 추정한 반품 예상액을 제외한 금액을 수익 인식

[꼬리질문 목적]
→ 사용자 상황이 이 QNA의 전제조건과 같은지 확인:
  "반품 시 대체 상품이 아닌 현금으로 환불하나요?"
  "반품 기간이 계약서에 명확히 명시되어 있나요?"
  "소매업(B2C) 판매인가요, 아니면 도매/B2B인가요?"
```

## 작업 방법

### Step 1: QNA 패턴 유형 분류
질의회신 데이터에서 자주 등장하는 패턴을 아래 유형으로 분류해주세요:

| 유형 | 설명 | 예시 |
|------|------|------|
| `반품_조건부판매` | 반품권이 포함된 판매 | 반품 가능 소매, 시범 사용 후 구매 |
| `라이선스_구분` | 라이선스의 접근권/사용권 구별 | 소프트웨어, 콘텐츠 |
| `설치_포함판매` | 재화 + 설치용역이 묶인 계약 | 기계 설치, 시스템 구축 |
| `고객충성제도` | 포인트·마일리지·적립금 | 멤버십, 리워드|
| `건설_진행률` | 기간에 걸친 수익 인식 진행률 산정 | 건설, 장기프로젝트 |
| `위탁_수탁` | 위탁자·수탁자 간 매출 귀속 | 입점판매, 위탁배송 |
| `비현금대가` | 물물교환, 설비 제공 등 | 광고 교환, 고객 제공 자재 |

### Step 2: 전제조건 (Preconditions) 추출
각 QNA 사례에서 **그 회계처리가 성립하기 위한 전제조건**을 추출하세요.

```
QNA ID: [질의회신 번호 또는 주제명]
회계처리 결론: [해당 QNA의 최종 답변]
전제조건 목록:
  - 조건1: [이 결론이 성립하기 위해 반드시 충족해야 하는 조건]
  - 조건2: ...
  - 조건3: ...
반대 케이스: [조건이 달라지면 결론이 어떻게 달라지는가]
```

### Step 3: 전제조건 → 꼬리질문 변환
추출된 전제조건을 사용자에게 물어보는 질문 형태로 변환하세요.

```python
QNA_MATCH_TREES = {
    "패턴_유형명": {
        "trigger_keywords": ["탐지 키워드"],
        "qna_premise_checklist": [
            {
                "condition": "전제조건 요약",
                "question": "사용자에게 물어볼 꼬리질문",
                "yes_path": "Yes일 때 해당 QNA 회계처리 적용 가능",
                "no_path": "No일 때 다른 케이스 검토 필요"
            },
        ],
        "related_qna_topics": ["관련 QNA 주제들"]
    }
}
```

## 출력 예시 (참고용)

```python
"반품_조건부판매": {
    "trigger_keywords": ["반품", "반품권", "반품 조건", "취소 옵션"],
    "qna_premise_checklist": [
        {
            "condition": "반품 시 환불 방식",
            "question": "반품 시 고객에게 현금 환불인가요, 대체 상품 제공인가요?",
            "yes_path": "현금 환불 → 변동대가(반품권) 처리 가능",
            "no_path": "대체 상품 → 교환 거래로 별도 검토 필요"
        },
        {
            "condition": "반품 기간 명확성",
            "question": "반품 가능 기간이 계약서에 구체적으로 명시되어 있나요?",
            "yes_path": "기간 명확 → 기댓값 추정 가능",
            "no_path": "기간 불명확 → 변동대가 제약 조건 검토 필요"
        },
        {
            "condition": "반품률 추정 가능성",
            "question": "과거 데이터 또는 유사 상품 데이터로 반품률 추정이 가능한가요?",
            "yes_path": "추정 가능 → 기댓값법 적용",
            "no_path": "추정 불가 → 가장 유력한 금액법 또는 전액 제약 검토"
        },
    ],
    "related_qna_topics": ["변동대가 추정", "반품자산 인식"]
},
```

## 주의사항

- QNA 체크리스트는 **"이 QNA가 내 상황에 적용 가능한가?"** 를 판단하기 위한 것입니다.
- 회계처리 결론 자체를 꼬리질문으로 만들지 마세요. 꼬리질문은 **전제조건 확인**입니다.
- 하나의 전제조건 = 하나의 꼬리질문으로 1:1 대응이 이상적입니다.
- yes_path / no_path를 명시하면, 나중에 대화 분기 로직 구현 시 사용할 수 있습니다.
- 불확실한 항목은 `# 검토 필요: [이유]` 주석으로 표시하세요.
