# A. 제품 전략

## 1. 범용 AI와의 차별화 — "이 챗봇을 쓸 이유 만들기"

### 배경

프로젝트 초기 목표는 K-IFRS 1115호 전문 AI 챗봇이었다. 기준서 원문을 학습시켜 회계감사인의 질문에 답하는 시스템을 구상했다.

### 문제

구현을 하는 중에 근본적 의문이 발생했다.

- **ChatGPT, Gemini 같은 범용 AI 대비 이 챗봇을 사용할 이유가 없다**
- 범용 AI도 회계 질문에 답할 수 있고, 더 다양한 주제를 커버한다
- 환각을 만들어 낸다는 위험이 있지만, 다양한 AI에서 검색기술을 통해 극복할 수 있다.

### 분석 과정

회계감사인이 실무에서 실제로 필요한 것이 무엇인지 추론해야 했다.

- **분석 1**: 단순 챗봇 AI가 아닌 다른 가치를 함께 제공해야 한다
  → 감사인이 기준서를 참조할 때의 상황을 가정했을 때, **"이 거래가 어떤 토픽에 해당하는지 파악 → 관련 문단·질의회신·감리사례를 수집 → 판단"** 할 것이라는 가설을 세웠다. 기준서 뿐만 아니라 다양한 문서 수집 과정 자체에 시간이 많이 소요될 것이라고 판단했다.

- **분석 2**: 기준서 본문이나 QNA·감리사례를 열람할 때, 각 항목의 **SUMMARY를 먼저 보여주면** 전체를 읽지 않고도 핵심을 빠르게 파악할 수 있을 것이라고 판단했다. 원문이 길고 전문 용어가 많은 회계 문서 특성상, 요약이 있으면 필요한 자료를 선별하는 시간을 줄일 수 있다.

- **결론**: 범용 AI가 제공하지 못하는 **"구조화된 근거 열람 + SUMMARY"** 기능을 1단계로 제공하고, AI 답변은 2단계로 배치한다.

### 해결

UX를 **4단계 State Machine**으로 설계했다.

```
[1단계] 홈 — 26개 토픽 목록
    ↓ 토픽 클릭
[2단계] 토픽 브라우즈 — 4탭 구조
    ├─ [본문·BC 탭] 기준서 원문 + 결론도출근거
    ├─ [적용사례 탭] IE 사례 (문단 단위 병합)
    ├─ [질의회신 탭] QNA 101건 중 해당 토픽 관련
    └─ [감리사례 탭] FSS·KICPA 감리지적사례 18건
    ↓ 질문 입력
[3단계] 근거 열람 — 검색 결과를 카테고리별 아코디언으로 구조화
    ↓
[4단계] AI 답변 — Split View (좌=근거, 우=답변+꼬리질문)
```

**핵심 설계 원칙: "근거 선행, AI 후행"**

| 항목 | 범용 AI (ChatGPT) | 이 시스템 |
|------|-------------------|----------|
| 사용 흐름 | 질문 → AI가 답변 | **토픽 선택 → 근거 열람 → AI에게 질문** |
| 근거 접근 | AI 답변에 포함(선택적) | **AI 없이도 4탭으로 즉시 열람** |
| 감리사례 | 제공 불가 | 토픽별 감리지적사례 연결 |
| 질의회신 | 학습 데이터에 포함 여부 불확실 | 4개 게시판 101건 직접 수집 |

### 결과

- 26개 토픽 × 4탭의 토픽 브라우즈 기능 구현
- 각 토픽에 SUMMARY를 제공하여 해당 토픽의 핵심 쟁점을 한눈에 파악 가능
- **AI 답변 없이도 가치를 제공하는 시스템**으로 차별화 달성

### 회고

- **배운 점**: AI 프로젝트에서 차별화는 "AI가 더 잘 답하는 것"이 아니라 **"AI 외에 어떤 가치를 함께 제공하느냐"**에서 나올 수 있다. 토픽 브라우즈는 AI 기능이 아니라 **데이터 큐레이션 + UX 설계**의 결과물이다.
- **다음에 같은 상황이면**: 사용자의 실제 워크플로우를 먼저 분석하고, AI뿐만이 아니더라도 도구적 가치에 가산점을 부여하여 설계한다.

# B. 데이터 파이프라인

## 2. PDF 파싱 한계 — 웹크롤링으로 전환

### 배경

K-IFRS 1115호 기준서(517페이지)를 AI가 검색할 수 있도록 가공해야 했다. 초기에는 **Upstage Document Parse API**를 이용한 PDF 파싱 방식을 선택했다.

- PDF에서 3,424개 요소를 자동 추출 (paragraph 2,061개, heading 410개, table 83개 등)
- 추출된 요소를 6개 섹션(본문, 부록A~C, 적용사례, 결론도출근거)으로 분리

### 문제

PDF 파싱 결과에서 **오류가 발견될 때마다 덧대기 패치**를 적용했으나, 볼 때마다 새로운 오류가 발생했다.

| 문제 | 설명 | 영향 |
|------|------|------|
| **HTML 태그 혼입** | PDF 파싱 후 HTML 태그(`<p>`, `<div>`, 인라인 스타일)가 잔류 | 임베딩 시 노이즈로 작용 |
| **표 구조 손실** | `colspan`/`rowspan` 미지원 → 복잡한 표(IE 분개표 등) 손상 | 적용사례 테이블 가독성 파괴 |
| **문단 번호 불일치** | 원본 PDF의 문단 번호와 파싱 결과의 매핑이 불안정 | 인용 시 잘못된 문단 참조 |
| **비결정성** | 같은 PDF를 다시 파싱하면 **다른 결과**가 생성됨 | 파이프라인 재현성 붕괴 |
| **섹션 경계 오류** | 헤더 인식 실패로 섹션 분리가 틀어지는 케이스 발생 | 수동 규칙으로 대응 → 한계 |

**사후 패치의 누적 문제**: 표 깨짐 → 정규식으로 복원 → 다른 표에서 새로운 깨짐 → 또 정규식 추가 → 수정 코드가 복잡해지면서 **사이드이펙트로 정상 문단까지 손상**

### 분석 과정

- **가설 1**: 파싱 옵션을 조정하면 해결된다 (split 전략, OCR 설정 등)
  → 검증: 여러 옵션을 시도했으나, 표 구조 손실과 비결정성은 PDF 파싱의 본질적 한계였다.

- **가설 2**: 다른 PDF 파싱 도구를 사용하면 해결된다
  → 검증: 도구를 바꿔도 PDF의 물리적 레이아웃 → 논리적 구조 변환 과정의 오류는 피할 수 없다.

- **가설 3**: 이미 구조화된 형태의 데이터 소스가 존재하는가?
  → 조사: **kifrs.com이 REST API로 기준서 원문을 제공**하고 있었다. HTML로 이미 구조화되어 있으며, 문단 번호(`documentId`, `paragraphNumber`)가 정확하게 매핑되어 있었다.

### 해결

PDF 파싱을 폐기하고, **kifrs.com REST API 기반 웹크롤링**으로 전환했다.

| 항목 | PDF 파싱 (Before) | 웹크롤링 (After) |
|------|-------------------|-----------------|
| 데이터 소스 | 로컬 PDF (517페이지) | kifrs.com REST API |
| 문단 번호 | 파싱 결과에서 추정 (불안정) | API가 직접 제공 (정확) |
| 표 구조 | colspan/rowspan 손실 | HTML → BeautifulSoup 직접 파싱 (커스터마이징 가능) |
| 재현성 | 같은 PDF에서 다른 결과 | **동일 API 호출 = 동일 결과** |
| 정제 로직 | 파싱 오류에 대한 사후 패치 | **3단계 정제 파이프라인** (HTML 파싱 → 섹션 정규화 → 문단 분리) |

전환 후 14개 스크립트를 3개 Phase로 구성한 전처리 파이프라인을 설계했다.

```
Phase 1: 수집 — [02] 기준서 크롤링 + [05] QNA 4개 게시판 크롤링
Phase 2: 가공 — [03] HTML→마크다운+가중치 청킹 → [04] 임베딩 → MongoDB 적재
Phase 3: 큐레이션 — [10] 토픽 파싱 → [12] 서머리 임베딩 → [99] 전수 검증
```

**PDF가 여전히 필요한 경우**: 감리지적사례(FSS 15건, KICPA 5건)는 API로 제공되지 않아 PDF에서 수동 추출했다. 이 경우에도 PDF를 직접 파싱하지 않고, **LLM이 읽고 구조화된 JSON으로 수동 변환**하는 방식을 택했다.

### 결과

- 문단 번호 정확도 100% (API 직접 제공)
- 파이프라인 재현성 100% (동일 입력 = 동일 출력)
- 각 Phase가 독립 실행 가능 → 데이터 소스 업데이트 시 해당 단계만 재실행

### 회고

- **배운 점**: **"PDF 전처리는 최후의 수단이다."** 이미 구조화된 데이터(API, HTML)가 존재하는지 먼저 조사해야 한다. PDF에서 표와 서식을 정확히 추출하는 것은 극히 어렵고, 파싱 오류에 대한 덧대기 패치는 **새로운 오류의 원인**이 된다.
- **다음에 같은 상황이면**: 데이터 수집 방식을 결정하기 전에, 대상 데이터가 API·HTML·CSV 등 이미 구조화된 형태로 존재하는지 **반드시 먼저 조사**한다. PDF 파싱은 다른 모든 방법이 불가능할 때에만 사용한다.

# C. AI 파이프라인

## 3. LLM 답변 분산 — 프롬프트 강화의 한계, PydanticAI 전환

### 배경

토픽 브라우즈(1단계)를 완성한 후, 2단계인 **AI 답변 기능**을 개발하기 시작했다. 초기 구현은 **LangGraph**(AI 워크플로우 프레임워크) + **LangChain** 기반이었다.

### 문제

AI 답변의 **분산이 크고**, 테스트 결과를 신뢰할 수 없었다.

- **동일한 질문에 매번 다른 형식의 답변**이 생성됨
- 어떤 때는 체크리스트를 제시하고, 어떤 때는 바로 결론을 내림
- 답변에 인용이 있을 때도 있고 없을 때도 있음
- 테스트를 3회 반복하면 3회 모두 다른 결과 → **정량 평가 자체가 불가능**

**직관적 대응: 프롬프트를 더 엄격하게 작성**

→ 결과: **프롬프트를 엄격하게 쓸수록 오히려 오류가 증가**했다.

| 시도 | 프롬프트 변경 | 결과 |
|------|-------------|------|
| 1차 | "반드시 체크리스트를 먼저 제시하라" 추가 | 체크리스트는 나오지만, 인용이 사라짐 |
| 2차 | "인용 문단을 반드시 포함하라" 추가 | 인용은 나오지만, 존재하지 않는 문단 번호 생성 (환각) |
| 3차 | 전체 프롬프트를 구조화하여 재작성 | 포맷은 개선되었으나, AI가 지시를 부분적으로만 따름 |
- 이후 여러번 시도를 했지만 시도할 때마다 결과가 달라져 품질을 테스트 하기 어려웠다.

### 분석 과정

프롬프트 튜닝의 한계를 분석했다.

- **가설 1**: 프롬프트가 부족해서 AI가 지시를 못 따르는 것이다
  → 검증: 프롬프트를 계속 강화했으나 **포맷 준수율이 안정되지 않음**. 프롬프트의 양이 아니라 **강제력의 문제**임을 확인.

- **가설 2**: LangGraph의 자유형 텍스트 출력 구조가 근본 원인이다
  → 분석: LangGraph에서 AI 출력은 **자유형 문자열**이다. 프롬프트로 "JSON으로 답하라"고 지시해도, AI가 이를 무시하면 파싱이 실패한다. 이때 **3단 폴백**(JSON 파싱 시도 → 실패 시 정규식 추출 → 실패 시 기본값)으로 대응하고 있었는데, 이 폴백 자체가 **답변 품질의 비결정성**을 만들고 있었다.

- **가설 3**: LangGraph/LangChain의 추상화 계층이 디버깅을 어렵게 만든다
  → 분석: LangGraph의 StateGraph는 노드 간 데이터 흐름을 프레임워크 내부에서 관리한다. 문제가 발생했을 때 **중간 상태를 확인하려면 프레임워크의 내부 구조를 파악**해야 했고, `print(state)` 한 줄로 확인할 수 있는 것이 아니었다. 프롬프트 수정 → 결과 확인 → 원인 추적의 반복 주기가 길어지면서, 문제의 원인이 프롬프트인지 프레임워크 동작인지 분리하기 어려웠다.

- **결론**: 프롬프트가 아니라 **설계 수준에서 출력 형식을 강제**해야 하며, 동시에 **디버깅이 용이한 단순한 구조**로 전환해야 한다.

### 해결

LangGraph/LangChain을 폐기하고, **PydanticAI**로 전환했다.

| 항목 | LangGraph (Before) | PydanticAI (After) |
|------|-------------------|-------------------|
| AI 출력 방식 | 자유형 텍스트 → 3단 폴백 파싱 | **구조화 출력** (양식 미충족 시 답변 거부) |
| 포맷 준수 강제 | 프롬프트에 의존 (무시 가능) | **코드 수준에서 강제** (무시 불가능) |
| 인용 누락 방어 | 없음 (누락되면 그대로 전달) | `result_validator`로 빈 인용 감지 → 자동 재시도 |
| 디버깅 | StateGraph 내부 추상화 → 중간 상태 추적 어려움 | `print(state)` 한 줄로 즉시 확인 |
| 파일 수 | graph.py, llm.py, grade.py 등 5개 | pipeline.py 1개 (200줄) |

**구조화 출력의 핵심**: AI가 반드시 채워야 하는 양식을 코드로 정의한다.

```python
class ClarifyOutput(BaseModel):
    selected_branches: list[str]   # 선택한 결론 분기 (필수)
    cited_paragraphs: list[str]    # 인용 문단 번호 (필수)
    follow_up_questions: list[str] # 꼬리질문
    conclusion_type: str           # TYPE 1(조건부) or TYPE 2(확정)
```

AI가 `cited_paragraphs`를 비워두면 **답변 자체가 거부**되고, `result_validator`가 자동으로 AI를 재호출한다 (최대 2회). 프롬프트 수준의 "부탁"이 아니라 **코드 수준의 강제**이다.

### 결과

| 지표 | LangGraph | PydanticAI |
|------|-----------|-----------|
| 포맷 준수율 | 불안정 | **100%** |
| 3단 폴백 | 필요 (JSON→regex→기본값) | **제거** |
| 인용 누락 | 간헐적 발생 | **자동 재시도로 0건** |
| 테스트 재현성 | 3회 반복 시 3회 다른 결과 | **형식은 100% 동일**, 내용만 변동 |

### 회고

- **배운 점**: AI의 출력 품질을 프롬프트로 통제하는 데는 한계가 있다. 프롬프트는 "부탁"이고, 구조화 출력은 "강제"이다. **정확성이 중요한 도메인에서는 강제가 필요하다.**
- **다음에 같은 상황이면**: LLM 출력의 형식이 중요한 프로젝트에서는 처음부터 구조화 출력을 지원하는 프레임워크(PydanticAI 등)를 선택한다. 프롬프트 튜닝에 시간을 투자하기 전에, 설계 수준의 해법이 있는지 먼저 확인한다.

## 4. 답변 내용의 비결정성 — Decision Tree로 알고리즘 강제

### 배경

PydanticAI 전환으로 **출력 형식**은 100% 통제하게 되었다. 하지만 새로운 문제가 드러났다.

### 문제

형식은 일관되지만, **내용이 매번 달랐다**.

- 같은 "본인 vs 대리인" 질문에 어떤 때는 "통제 여부를 확인하세요", 어떤 때는 "가격결정권을 확인하세요"라고 답함
- 인용하는 문단이 매번 다름 (문단 B35를 인용할 때도, B37을 인용할 때도 있음)
- 결론 분기를 AI가 **자의적으로 생성** — 기준서에 없는 분기가 등장하는 경우도 발생

**근본 원인**: 추론형 LLM(Gemini)의 비결정성. 추론 과정에서 매번 다른 경로를 탐색하기 때문에, 동일 입력에도 다른 출력이 나온다. PydanticAI가 강제하는 것은 **"어떤 칸을 채우느냐"**이지, **"칸에 무엇을 쓰느냐"**는 여전히 AI의 자유이다.

### 분석 과정

- **가설 1**: 프롬프트에 구체적 체크리스트를 명시하면 해결된다
  → 검증: 26개 토픽 모두에 대한 체크리스트를 프롬프트에 넣으면 토큰 수가 초과한다. 토픽별로 다른 프롬프트를 동적으로 생성해야 한다.

- **가설 2**: 토픽별 사전 정의된 체크리스트·결론 분기·참조 문서를 만들어 AI에게 강제하면 된다
  → 이것이 **Decision Tree** 개념이다.

- **추가 검증**: AI 교차검증으로 Decision Tree의 품질을 보장
  → Gemini, Claude, NotebookLM 3개 AI에 동일 토픽에 대해 독립적으로 체크리스트·결론 분기를 생성시킨 후, **공통 항목은 채택하고 불일치 항목은 기준서 원문으로 검증**.

### 해결

K-IFRS 1115호의 판단 단위를 **26개 토픽으로 분류**하고, 각 토픽에 **7개 섹션의 Decision Tree**를 사전 작성했다.

| # | 섹션 | 역할 |
|---|------|------|
| 1 | **routing** | 이 토픽에 해당하는 키워드 + 판단 목표 |
| 2 | **checklist** | AI가 사용자에게 확인할 질문 목록 (고정) |
| 3 | **conclusion_guide** | AI가 선택할 수 있는 결론 분기 (고정, 신규 분기 생성 금지) |
| 4 | **precedents** | 분기별 참조할 적용사례·질의회신 문서 ID |
| 5 | **red_flags** | 분기별 감리지적사례 경고 |
| 6 | **calculation_formula** | 해당 시 계산 공식 |
| 7 | **critical_factors** | 양방향 분기 토픽의 핵심 판단 요소 |

**작동 방식**: 사용자 질문이 입력되면 3중 매칭(키워드 + LLM 추론 + 임베딩 유사도)으로 토픽을 식별하고, 해당 토픽의 Decision Tree를 AI 프롬프트에 동적 주입한다. AI는 사전 정의된 체크리스트를 확인하고, 사전 정의된 결론 분기 중 하나를 선택하며, 사전 배정된 문서만 인용할 수 있다.

**환각 방지 5단계 레이어 완성**:

```
[Layer 1] Decision Tree 큐레이션 — 체크리스트·결론 분기 사전 정의
[Layer 2] PydanticAI 구조화 출력 — cited_paragraphs, selected_branches 필수
[Layer 3] result_validator — 빈 인용 시 자동 재시도 (최대 2회)
[Layer 4] reasoning_guard — 논리 뒤집음 방지, 새 분기 생성 금지
[Layer 5] Precedents 직접 주입 — 큐레이션 문서를 검색 엔진 우회하여 직접 전달
```

### 결과

Decision Tree 도입 전후의 품질 변화:

| 지표 | Before (PydanticAI만) | After (+ Decision Tree) | 개선폭 |
|------|----------------------|------------------------|--------|
| 검색 적중률 | 36% | **76%** | **+40%p** |
| 꼬리질문 일관성 | 60% | **100%** | **+40%p** |
| 환각 발생 | 간헐적 | **0건** (골든 테스트 159회) | **100% 달성** |
| 포맷 준수율 | 100% | 100% | 유지 |

> 검증 체계: 모델 비교 218회 + 골든 테스트 159회 = 총 377회. 환각 0건과 적중률 76%는 골든 테스트 159회 기준이며, 36%는 프로덕션 통합 직후(3차 테스트) 수치이다.

특히 **검색 적중률 +40%p**는 프로젝트 전체에서 단일 최대 개선이다. Decision Tree의 `precedents` 섹션에 배정된 문서 ID를 MongoDB에서 직접 조회(핀포인트 검색)하는 구조가 핵심이었다. 이 과정에서 발견한 **파싱 3중 Gap**(소괄호 미파싱, ID 축약 불일치, 섹션 미수집)을 동시 수정하여 달성했다.

### 회고

- **배운 점**: **"AI의 자유도를 제한할수록 신뢰도가 올라간다."** 이 프로젝트에서 가장 큰 발견이다. 프롬프트 강화 → PydanticAI(형식 강제) → Decision Tree(내용 강제)로 점진적으로 AI의 자유도를 줄여나갔고, 줄일 때마다 품질이 올라갔다.
- **다음에 같은 상황이면**: 정확성이 중요한 도메인에서는 처음부터 AI에게 "자유롭게 답하라"가 아니라 **"이 선택지 중에서 골라라"** 구조를 설계한다. 자유도 제한이 품질 향상에 직접적으로 기여한다.

## 5. Git 미숙으로 작업 소실 — 에이전트와 규칙 체계 수립

### 배경

이 프로젝트는 Git을 처음 실전에서 사용한 프로젝트이다. `develop`/`main` 두 브랜치만 운영하면서 개발을 진행했다.

### 문제

**`git add` 없이 `git switch`로 브랜치를 전환하여 작업 내용이 전체 소실**되었다. 복구 불가능했다.

```bash
git switch main          # ← unstaged 변경사항이 모두 사라짐
# 작업 내용 전체 소실 — 복구 불가
```

이 사고 이후 추가로 발견된 문제:

| 사고 | 원인 |
|------|------|
| tmp 파일 29개가 커밋에 포함 | `.gitignore`에 `*.tmp.*` 패턴 누락 |
| 32MB 바이너리(PDF/HWP 34개) 커밋 | `data/` 폴더를 `.gitignore`에 미등록 |
| API 키 노출 위험 | `.env`만 제외하고 `.claude/settings.local.json` 등 미제외 |

### 분석 과정

GIT 실력이 부족한것과, **프로젝트 시작 시 기본 설정을 하지 않은 것**이 근본 원인이었다.

- `.gitignore`가 2줄(`.env`, `__pycache__/`)만 있었다
- 커밋 전 `git status` 확인 습관이 없었다
- 브랜치 전환 전 stash/commit을 하지 않았다

### 해결

Claude Code 에이전트와 함께 Git 규칙 체계를 수립했다.

1. **`.gitignore` 체계화** (2줄 → 24줄) — 환경변수, 가상환경, 캐시, 대용량 데이터, 임시 파일, IDE 설정 등 포괄적 규칙 수립
2. **`git add → status → commit` 3단계 습관** — 브랜치 전환 전 반드시 커밋 또는 stash
3. **CLAUDE.md에 Git 규칙 명문화** — 에이전트가 force push, reset --hard 등 파괴적 명령을 실행하지 못하도록 방어 규칙 설정

> 구체적인 `.gitignore` 예시, git 명령어 패턴, 정리 상세는 [6_LESSONS-LEARNED.md §5 Git 관리 교훈](6_LESSONS-LEARNED.md#5-git-관리-교훈--첫-프로젝트에서-배운-것)을 참조한다.

### 결과

V1 완성 시점에서 기존 불필요 파일을 정리했다.

| 정리 항목 | Before | After |
|-----------|--------|-------|
| 추적 파일 수 | 241개 | **156개** |
| `.gitignore` 규칙 | 2줄 | **24줄** |
| tmp 파일 | 29개 | **0개** |
| 바이너리 데이터 | 32MB (PDF/HWP 34개) | **0개** (`git rm --cached`) |
| 테스트 결과 JSON | 24개 | **0개** |

### 회고

- **배운 점**: `.gitignore`는 프로젝트 **첫 `git init` 직후에** 작성해야 한다. 나중에 정리하는 것은 이미 커밋된 파일을 `git rm --cached`로 하나씩 제거하는 번거로운 작업이다. 이 정리 작업 자체가 `.gitignore`를 처음에 제대로 설정했으면 불필요했다.
- **다음에 같은 상황이면**: 이 프로젝트에서 수립한 Git 규칙은 다음 프로젝트에서도 그대로 적용하고 있다. 특히 에이전트와 협업할 때, CLAUDE.md에 Git 규칙을 명문화하면 에이전트가 파괴적 명령을 실행하는 사고를 예방할 수 있다.
-> 실제로 다음 프로젝트부터 명문화된 Git 규칙을 통해 안정적으로 운영이 가능했다.

## 6. 문서 부재로 인한 반복 디버깅 — docs 체계 수립

### 배경

프로젝트 전반기에는 별도의 문서 관리 없이 개발을 진행했다. 디버깅 교훈, 의사결정 근거, 프로젝트 컨벤션 등을 기록하지 않았다.

### 문제

**같은 문제가 반복 발생하고, 매번 처음부터 디버깅**하는 상황이 계속되었다.

- 이전에 해결한 문제를 다시 마주쳤을 때, 당시 원인과 해결책을 기억하지 못해 동일한 삽질을 반복
- AI 에이전트와 협업할 때, 에이전트가 프로젝트의 컨텍스트(기술 스택, 컨벤션, 알려진 이슈)를 모르므로 **매 대화마다 같은 설명을 반복**해야 했다
- 의사결정의 근거가 기록되지 않아, 과거에 왜 그런 선택을 했는지 알 수 없었다. 결과적으로 **이미 검증하고 폐기한 방안을 다시 시도**하는 경우가 발생

### 분석 과정

- **분석 1**: 디버깅 시간의 상당 부분이 "문제 해결" 자체가 아니라 **"이전에 이걸 어떻게 해결했는지 다시 파악하는 것"**에 소요되고 있었다. 기록이 있었다면 즉시 참조하고 넘어갈 수 있는 문제였다.

- **분석 2**: AI 에이전트는 대화가 끝나면 컨텍스트가 초기화된다. 프로젝트의 기술 스택, 컨벤션, 알려진 함정을 **파일로 남겨두면 에이전트가 매번 읽고 시작**할 수 있다. 사람을 위한 문서와 에이전트를 위한 문서를 모두 갖추면 협업 효율이 올라간다.

- **결론**: `docs/` 폴더에 체계적 문서를 구축하고, 작업 시작 전 반드시 참조하는 워크플로우를 수립해야 한다.

### 해결

프로젝트 후반기에 4종의 핵심 문서를 작성했다.

| 문서 | 역할 | 주요 독자 |
|------|------|----------|
| `CLAUDE.md` | 프로젝트 컨텍스트, 기술 스택, 컨벤션, 명령어 | **에이전트** (매 대화 시작 시 자동 로드) |
| `docs/debugging.md` | 디버깅 교훈 40건 (문제·원인·해결·교훈) | 에이전트 + 본인 |
| `docs/DECISION.md` | 의사결정 기록 33건 (ADR: 결정·맥락·근거) | 본인 (과거 판단 추적) |
| `docs/STREAMLIT.md` | Streamlit 고유 제약과 우회법 | 에이전트 + 본인 |

**CLAUDE.md의 워크플로우 강제**:

```markdown
### 작업 전 (필수)
- docs 파일에서 필요한 문서를 읽고 작업 시작
1. PROJECT_OVERVIEW.md : 프로젝트 전체 개요
2. debugging.md : 디버깅 교훈
3. STREAMLIT.md : Streamlit 규칙
4. DECISION.md : 과거 주요 의사결정

### 작업 후 (필수)
- docs 폴더에서 참조했던 파일 업데이트
```

에이전트가 작업을 시작할 때 반드시 문서를 읽고, 작업이 끝나면 새로운 교훈을 문서에 추가하는 순환 구조를 만들었다.

### 결과

- 디버깅 교훈 40건이 누적되면서, 동일 문제 재발 시 **즉시 참조하여 해결** 가능
- 에이전트가 CLAUDE.md를 읽고 시작하므로, 기술 스택·컨벤션·알려진 함정을 **매번 설명하지 않아도** 됨
- 의사결정 기록 33건(ADR)으로, 과거에 폐기한 방안을 다시 시도하는 낭비가 제거됨

### 회고

- **배운 점**: 문서화는 "프로젝트 완료 후 정리"가 아니라 **"개발 과정 중 실시간 기록"**이어야 한다. 후반기에 작성했기 때문에 전반기의 교훈 상당수는 기억에 의존하여 복원해야 했다. 처음부터 기록했다면 더 정확하고 완전했을 것이다.
- **다음에 같은 상황이면**: 다음 프로젝트에서는 **첫 커밋부터** CLAUDE.md + docs 구조를 세팅하고 시작했다. 실제로 에이전트 협업 효율이 눈에 띄게 향상되었고, 디버깅 반복이 크게 줄었다.
