# 05. LLM 선정·듀얼 라우팅

> 작성일: 2026-07-02 / 작성자: AI 초안 (예시 완성본 — 사용자 검증·본인 언어로 수정 필요) / AI 검증: —
> ⚠️ 이 파일은 형식 예시다. 아래 내용을 본인이 `app/test/model_comparison/` 결과와 대조하고 본인 문장으로 고쳐야 "내 것"이 된다.

## ① Painpoint — 감사 실무의 어떤 문제인가

수익 인식 질문은 두 종류의 능력을 동시에 요구한다.

- **회계 추론**: "본인인가 대리인인가" 같은 판단 논리 (기준서 문단 적용)
- **산술 정확도**: 진행률·거래가격 배분 같은 계산 (감사에서 1원이라도 틀리면 신뢰 상실)

하나의 LLM으로 두 능력을 모두 확보할 수 있는지가 검증되지 않은 상태였다.

## ② AI 활용 해결 — 왜 AI로, 어떤 방식으로 푸는가

- AI 없이는: 사람이 기준서를 뒤져 판단하고 엑셀로 계산 — 도구 자체가 성립 안 함.
- AI를 그냥 쓰면: 어떤 모델이 회계 질문에 강한지 벤더 벤치마크(수학·코딩 중심)로는 알 수 없다. 회계 도메인 벤치마크는 존재하지 않았다.
- 채택한 구도: **자체 평가셋으로 모델별 A/B 테스트를 돌려 데이터로 선정**하고, 단일 모델로 양립이 안 되면 질문 유형별로 모델을 나누는 **듀얼 라우팅**을 설계한다.

## ③ 기술 선택 — 무엇을 검토했고 왜 이것인가

| 후보              | 회계 추론  | 산술 정확도 | 선택 여부와 이유                   |
| ----------------- | ---------- | ----------- | ---------------------------------- |
| Gemini Flash      | 0.81 (1위) | 20~40%      | ✅ 일반·상황 질문 전담 (추론 1위)  |
| GPT-4.1-mini      | 0.71 (4위) | 100%        | ✅ 계산 질문 전담 (산술 유일 100%) |
| Claude 3.5 Sonnet | 0.76 (2위) | 67%         | ✕ 추론 2위지만 산술 불충분         |
| GPT-4o            | 0.73 (3위) | 33%         | ✕ 양쪽 모두 어중간                 |
| 기타 3개 모델     | 0.41~0.69  | 0~33%       | ✕ 1차 라운드 탈락 포함             |

(수치 출처: `app/test/model_comparison/` 실행 결과, 이력서 project_kifrs.html Quality Verification 탭)

- 결정적 근거: **회계 추론 1위 모델과 산술 1위 모델이 다르다** → 단일 모델 포기, 듀얼 라우팅 채택.
- 라우팅 방식: `_needs_calculation()` 판정으로 계산 질문만 GPT-4.1-mini로 분기 (구현: `app/nodes/generate.py`, 개요: `docs/PROJECT_OVERVIEW.md` §4).

## ④ 검증 — 수치·증거

4개 라운드, 총 218회 호출 (출처: 이력서 Quality Verification 탭, `app/test/model_comparison/run.py`, `run_production.py`):

| 라운드 | 대상     | 호출  | 결과                              |
| ------ | -------- | ----- | --------------------------------- |
| 1차    | 7개 모델 | 28회  | 초기 선별 — 2개 탈락              |
| 2차    | 6개 모델 | 150회 | 정밀 비교 — 듀얼 LLM 확정         |
| 3차    | 듀얼 2종 | 15회  | 프로덕션 통합 — 오라우팅 발견     |
| 4차    | 듀얼 3종 | 25회  | 개선 확인 — 평균 점수 0.72 → 0.82 |

부수 발견: 계산 모델(GPT-4.1-mini)에 사고과정(CoT) 유도 문구를 앞에 붙이면 오히려 산술 정확도가 떨어져 제거함 (출처: CLAUDE.md 코드 컨벤션).

## ⑤ 한계 & 다시 한다면

- 트레이드오프: 모델 2개 운영 = API 비용·장애 지점 2배. 라우팅 오판 시 약한 모델이 답한다 (3차에서 실제 발생).
- 평가셋이 자체 제작 소규모(라운드당 수십 문항)라 통계적 신뢰구간은 좁지 않다 — "경향 확인" 수준임을 인정해야 함.
- 다시 한다면: 라우팅 판정 자체의 정확도(계산 질문을 계산으로 분류한 비율)를 별도 지표로 측정했을 것.

## ⑥ 예상 면접 질문 & 내 답변

- Q1: 모델을 어떻게 골랐나요?
  - A: 벤더 벤치마크는 회계 도메인을 반영하지 않아서, 수익 인식 질문으로 자체 평가셋을 만들어 7개 모델을 218회 테스트했습니다. 추론 1위와 산술 1위가 달라서 둘을 라우팅으로 결합했습니다.
- Q2: 그냥 제일 좋은 모델 하나 쓰면 안 되나요?
  - A: 추론 1위 모델의 산술 정확도가 20~40%였습니다. 감사 도구에서 계산 오류는 치명적이라, 계산 질문만 산술 100% 모델로 분기했습니다.
- Q3: 라우팅이 틀리면요?
  - A: 실제로 3차 테스트에서 오라우팅을 발견해 판정 로직을 고쳤고, 4차에서 평균 점수가 0.72에서 0.82로 오른 것을 확인했습니다. (→ 판정 로직이 구체적으로 어떻게 고쳐졌는지는 본인이 재확인 필요)
